# IU Requirement Mapping

| IU Requirement | Project Implementation | Explanation for Presentation |
|---|---|---|
| Data ingestion | Simulated sensor measurements are generated in `data/sensor_generator.py`. Training data is created with `data/create_training_data.py`, while live-like measurements are continuously produced by `data/sensor_stream.py`. | The system ingests simulated IoT sensor data containing temperature, humidity, and sound volume. This replaces real factory sensor input and allows the pipeline to be tested locally and reproducibly. |
| Stream processing | The continuous data stream is simulated in `data/sensor_stream.py`. The script generates sensor measurements at configurable time intervals and sends them to the API endpoint `POST /predict`. | The stream application represents continuous sensor measurements in a production environment. Instead of processing a static dataset only once, the system repeatedly sends new measurements to the prediction service, which reflects the stream processing focus of the task. |
| Model training | The anomaly detection model is trained in `model/train_model.py` using the simulated training dataset `data/training_sensor_data.csv`. The script trains an Isolation Forest model, evaluates it with a confusion matrix and classification report, and saves the trained model as `model/anomaly_model.joblib`. | The model training step prepares a reusable anomaly detection model. A simple Isolation Forest was chosen because the project focuses on production integration rather than maximum model complexity. The trained model is stored as an artifact so that it can later be loaded by the API without retraining. |
| RESTful API | The REST API is implemented in `api/app.py` using FastAPI. It provides a `POST /predict` endpoint that accepts sensor measurements as JSON input and returns a prediction with `predicted_anomaly` and `anomaly_score`. It also provides a `/health` endpoint to check whether the service and model are available. | The REST API exposes the trained model through a standardized interface. This allows other applications, such as the simulated sensor stream, to send sensor measurements and receive predictions without directly accessing the model code. |
| Model as a service | The trained model is saved as `model/anomaly_model.joblib` and loaded by the FastAPI application in `api/app.py`. The API uses this loaded model to generate predictions for incoming sensor measurements. | The model is separated from the training process and served through an API. This means the model does not need to be retrained for every request. Instead, it is trained once, stored as an artifact, loaded by the service, and used for real-time inference. |
| Monitoring | Predictions are logged from `data/sensor_stream.py` into `monitoring/prediction_log.csv`. The script `monitoring/analyze_predictions.py` reads this log file and prints a monitoring summary including total measurements, predicted anomalies, simulated anomalies, average anomaly score, and highest anomaly score. | The monitoring component makes model behavior observable over time. Instead of only showing predictions in the terminal, the system stores each prediction with timestamp, input values, anomaly flag, and anomaly score. This allows later inspection of model behavior and supports reliable execution of the predictive service. |
| Maintainability | The project is organized into clear modules: `api/`, `data/`, `model/`, `monitoring/`, and `docs/`. Each file has a focused responsibility, and dependencies are documented in `requirements.txt`. | The modular structure makes the system easier to understand, maintain, and extend. Developers can update the sensor simulation, model training, API, or monitoring logic independently without changing unrelated parts of the project. |
| Scalability | The system separates data generation, model training, API serving, streaming, and monitoring into different components. The prediction service is exposed through FastAPI and could be deployed independently from the stream application. | The modular architecture supports scalability because the API service, stream producer, model artifact, and monitoring layer are not tightly coupled. In a real production setup, the API could be containerized, replicated, or moved to cloud infrastructure, while the sensor stream could be replaced by real IoT data sources. |
| Adaptability to new data | New simulated training data can be generated with `data/create_training_data.py`, and the model can be retrained with `model/train_model.py`. The updated model is saved again as `model/anomaly_model.joblib` and can be loaded by the API. | The system can be adapted to new data by regenerating or replacing the training dataset and retraining the model. This allows the model to be updated when sensor patterns change over time, for example due to machine wear, changed production conditions, or data drift. |
| Reproducibility / Code link | The project is available in a public GitHub repository: `https://github.com/tom-roth/anomaly-detection-iot`. The `README.md` explains the project goal, architecture, model choice, setup steps, and execution commands. Dependencies are listed in `requirements.txt`. | The public repository allows the audience to access and reproduce the project. By creating a virtual environment and installing the dependencies from `requirements.txt`, users can generate training data, train the model, start the API, run the sensor stream, and analyze logged predictions. |
